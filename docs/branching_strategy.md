# Стратегия ветвления

Проект использует упрощённый вариант Git Flow. Все стабильные релизы хранятся в ветке `main`, активная разработка ведётся в `develop`, новые возможности создаются в ветках `feature/*`, а подготовка релизов выполняется в ветках `release/*`.

## Основные ветки
- **`main`** — только стабильные релизы. Содержит проверенный код, который готов к деплою.
- **`develop`** — интеграционная ветка для ежедневной разработки. В неё вливаются готовые фичи и фиксы перед подготовкой релизов.
- **`feature/<название>`** — ветки для разработки отдельных задач. Название соответствует задаче или тикету.
- **`release/<версия>`** — краткоживущие ветки для подготовки релизов (staging, фиксы). После релиза вливаются в `main` и `develop`.

## Правила доступа
- Прямой пуш в `main` и `develop` запрещён. Все изменения вносятся через pull request (PR).
- Создавать и пушить ветки `feature/*` и `release/*` могут разработчики, ответственные за задачу или релиз.
- Ответственность за мёрдж PR в `main` и `release/*` несут только мейнтейнеры (лиды).

## Требования к pull request
1. **Ревью.** Минимум один аппрув от другого разработчика (для критических изменений — два).
2. **Проверки.** Все автоматические тесты и линтеры в CI должны завершиться успешно. При отсутствии автоматических проверок — запускайте локально и прикладывайте результат.
3. **Чистая история.** Перед мёрджем убедитесь, что ветка синхронизирована с целевой (rebase/merge).
4. **Описание.** В PR указывайте цель изменений, связанную задачу и список ключевых правок.
5. **Теги.** После закрытия релизного PR в `main` создаётся тег версии (например, `v1.2.0`).
